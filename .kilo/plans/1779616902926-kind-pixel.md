# Multiplication Town - Battle System & Stage Progression Bug Report

## 1. BATTLE SYSTEM BUGS

### 1.1 Timer Drift & Interval Management (Line 5205-5241)
**Severity: Medium**

The battle timer uses `setInterval(fn, 100)` and subtracts `0.1` each tick. JavaScript timers drift over time due to single-threaded execution. More critically, if `generateBattleQuestion()` is called multiple times before a previous interval is cleared, multiple intervals run concurrently.

```javascript
// Line 5205-5211: Safety check exists but insufficient
if (!battleState.inBattle) {
  clearInterval(battleState.battleTimerInterval);
  battleState.battleTimerInterval = null;
  return;
}
```

**Issue**: `battleDefeat()` sets `inBattle = false` synchronously, but other code paths may not properly guard against concurrent intervals.

---

### 1.2 Missing Null Check in `battleTimeUp()` (Line 5279, 5288)
**Severity: High**

```javascript
document.getElementById('battleOptions').querySelectorAll('.battle-option').forEach(b => b.disabled = true);
```

In input mode (`answerMode === 'input'`), there are no `.battle-option` elements — only an `<input>` element. This silently fails, leaving no disabled state, allowing double-submission.

---

### 1.3 `showMonsterIntent` Function Name Collision (Line 6731 vs 7117)
**Severity: High**

Two functions share the same name:
- Line 6731: `showMonsterIntent(action)` — takes monster action object
- Line 7117: `showMonsterIntent(type, text)` — takes type and text strings

The second definition (line 7117) overwrites the first. When `prepareMonsterTurn()` calls `showMonsterIntent(action)` at line 6822, it actually calls the wrong function signature, causing undefined behavior.

```javascript
setTimeout(() => {
```

After `heroTakeDamage()` triggers defeat, `battleDefeat()` sets `inBattle = false`. The subsequent `prepareMonsterTurn()` call then has no effect, and the setTimeout checks fail. The monster never gets its turn.

### 1.5 Answer Input Never Cleared Between Questions (Line 5366-5372)

### 1.6 Shield Gain Logic Has No Upper Bound Check (Line 5415-5418)
**Severity: Low**

```javascript
if (battleState.combo >= 3) {
  battleState.shield = Math.min(battleState.shield + 1, 5);
```

This triggers every time combo >= 3. If combo goes 3->4->5, shield gains on 3 and 5 but not 4. However, the `Math.min(..., 5)` cap is correct. The issue is the condition fires on every question answered correctly once combo >= 3, which is actually the intended design.

**Actual bug**: The logic says `combo >= 3` but combo is reset to 0 on wrong answers. So this should trigger on every correct answer while combo >= 3, which could grant multiple shields rapidly.

## 2. STAGE PROGRESSION SYSTEM BUGS

### 2.2 `completeLevel` Never Receives Calculated Stars (Line 6115-6137)
**Severity: High**

```javascript
  level.completed = true;

```

The `stars` parameter is passed in from `monsterDefeated()` but the actual star rating is calculated using `calculateStarRating()` and stored in a local variable — it was never passed to `completeLevel()`. Looking at `monsterDefeated()` line 7281:

```javascript
const stars = calculateStarRating(accuracy, timeLeft, totalTime);
```

But then on line 7286:
```javascript
unlockNextStage(battleState.currentStage);
```

`unlockNextStage()` doesn't exist at that scope — it's a different function. And `completeLevel()` is never called.

**Severity: High**

The `worlds` array defines level IDs as 1-15 across worlds:

- veteran levels: 6-10

But the `stages` array uses IDs 1-9. When `selectStageForBattle()` at line 5930 calls `getStageData(stageId)` with a world level ID, it may return the wrong stage or the first stage.

```javascript
function getStageData(stageId) {

}
```

## 3. EDGE CASES & VULNERABILITIES

### 3.1 Division by Zero in Accuracy Calculation
If `totalAnswered = correct + wrong = 0`, then `accuracy = correct / 0` in `monsterDefeated()` line 7278. Should check for zero.

### 3.2 Negative Numbers Not Validated
`num1` and `num2` are set via random functions but never validated to be positive. If `selectedLevel` becomes negative (e.g., `-1` for mixed is intentional but `-2` could cause issues).

### 3.3 localStorage Corruption
Multiple JSON.parse operations on localStorage data could throw if data is corrupted, causing silent failures.

### 3.4 Race Condition in Achievement System
`checkAchievements()` reads `maxComboEver` and compares with `battleState.combo`, but these are separate state mutations without atomic guarantees.

### 3.5 Timer Modifier Multiplication Could Exceed Limits
Line 5192: `baseTime = baseTime * stage.timerMod` — if `timerMod` is undefined or 0, timer could become 0 or broken.
