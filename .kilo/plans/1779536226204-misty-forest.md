# UX/UI Audit: Battle Mechanism & Answer Stage

## Audit Scope
- **Battle Mechanism**: Arena layout, hero/monster interactions, HP/timer displays, combo system, reward flows
- **Answer Stage**: Question display, input methods (numpad vs choice), feedback animations, timing system

---

## Critical Issues (Must Fix)

### 1. `numpadInput()` Has Debug Alerts (Lines 5225-5245)
The number pad input function contains `alert()` debugging statements that interrupt gameplay:
```javascript
function numpadInput(num) {
  alert('numpadInput: ' + num);  // Line 5225
  // ...
  alert('Input not found, regenerating...');  // Line 5229
  // ...
  alert('Still no input!');  // Line 5243
```
**Impact**: Completely breaks mobile/touch UX. Every tap on numpad shows an alert.
**Fix**: Remove all `alert()` calls from this function.

### 2. CSS Malformation in `.battle-header` (Lines 2081-2094)
The battle header has a broken CSS structure:
```css
.battle-header {
  display: flex;
  align-items: center;
  justify-content: space-between;
  width: 100%;
  padding: 8px 16px;
  box-sizing: border-box;
}  // <-- Line 2088 closes the rule
  background: rgba(0, 20, 15, 0.85);  // This is orphaned!
  border-radius: 12px;
  border: 2px solid #3a8f6f;
  margin-bottom: 8px;
  z-index: 2;
```
**Impact**: Styles after closing brace are ignored. Timer bar positioning affected.
**Fix**: Remove the closing brace at line 2088 or restructure the CSS block.

### 3. Duplicate `updateComboDisplay()` Functions (Lines 4950-4997, 6629-6669)
Two completely separate functions with same name exist:
- Lines 4950-4997: Uses `#comboCount`, `#comboSkill`
- Lines 6629-6669: Uses `countEl`, `skillEl`, `#comboSkill`
**Impact**: Confusing code, potential inconsistent behavior. The second definition overwrites the first.
**Fix**: Consolidate into single function with proper logic.

## Major Issues (Should Fix)

### 4. Input Mode Shows Both NumPad AND Text Input
When `answerMode === 'input'`:
- Shows `<input type="tel">` field
- Also shows number pad buttons below
- But numpadInput has broken alerts
- User interaction is confusing - tap numpad or keyboard?
**Fix**: Either hide numpad in input mode, or wire numpad to properly fill the input field without alerts.

### 5. Wrong Answer Feedback Inconsistency
- Input mode: Shows correct answer hint with border color change on input
- Choice mode: Highlights correct button but no explicit "correct answer" text
**Fix**: Add consistent "✓ 正確答案：X" feedback for both modes.

### 6. Battle Timer Cleanup on Mode Switch
When calling `setAnswerMode()` during active battle:
```javascript
function setAnswerMode(mode) {
  battleState.answerMode = mode;
  // If in battle, regenerates question
  if (battleState.active && battleState.inBattle) {
    generateBattleQuestion();  // Creates NEW timer interval
  }
}
```
The previous timer interval is not cleared before starting new one.
**Fix**: Clear existing `battleState.battleTimerInterval` before generating new question.

### 7. Answer Input Not Properly Reset Between Questions
After wrong answer with input mode:
- `input.disabled = true` is set
- Correct answer hint shown
- 1500ms delay then `generateBattleQuestion()` called
- But the input field might retain old value or state issues
**Fix**: Ensure full cleanup of input element before next question.

### 8. `closeBattleOverlay()` Doesn't Reset Battle State
```javascript
function closeBattleOverlay() {
  // clears timer interval
  switchMainSection('learn');
  battleState.active = false;
  battleState.inBattle = false;
  // Does NOT call resetBattleState()
}
```
**Impact**: Returning to battle later may have stale combo, gold, monster state.
**Fix**: Call `resetBattleState()` in `closeBattleOverlay()`.

## Moderate Issues (Nice to Fix)

### 9. Shield Display Not Visually Integrated
- Shield count shows in `battle-attributes` bar (line 2933)
- No visual indicator on hero when shield is active
- Shield prevents damage but user can't tell remaining shield mid-battle
**Fix**: Add shield icon near hero HP bar or on hero character itself.

### 10. HP Bar Color States Inconsistent
- Monster HP uses `.hp-bar-fill` with `high`/`medium`/`low` classes
- Hero HP uses `.hero-hp-fill` with only `low` class for red
**Impact**: Different visual treatment for player vs enemy HP feedback.
**Fix**: Use consistent class naming and color thresholds.

### 11. Stage Completion Triggers on Single Monster Defeat
In `monsterDefeated()`:
```javascript
if (battleState.currentStage) {
  // ... stage.completed = true after ONE monster
  battleState.inBattle = false;
  showStageVictory(stars, stage);
  return;
}
```
**Impact**: Stage feels too easy - one monster = complete stage. Unlike classic RPG where you'd fight multiple waves.
**Fix**: Either require defeating 2-3 monsters per stage, or make stage HP larger with multiple phases.

### 12. Adaptive Difficulty Lacks Visual Feedback
The system adjusts difficulty but user sees no clear indication:
- No badge/indicator showing "Easy Mode" or "Hard Mode" active
- Timer bar shows timing changes but not labeled as adaptive
**Fix**: Add visible difficulty indicator that changes color/style based on current adaptive state.

### 13. Question Type Badge Overwrites InnerHTML
```javascript
document.getElementById('battleQuestion').innerHTML =
  typeBadge + '<div style="margin-top:8px;font-size:1.5rem;">' + questionText + '</div>';
```
**Impact**: Any child elements previously in that container are removed. Could affect accessibility or future extensions.
**Fix**: Use proper DOM manipulation or template approach.

### 14. Wrong Answer Counter Shows But Input Stays Visible
After wrong answer in choice mode, buttons are disabled but:
- Input field remains visible if was in input mode
- The "timeout" or "wrong" state visual feedback doesn't clearly indicate to user they should wait
**Fix**: Add clear visual state indication (overlay, dimming, or message) during feedback delay.

## Minor Issues (Minor UX Polish)

### 15. Number Pad Clear Button Label
`onclick="numpadInput('C')"` - Button shows "C" but could be more intuitive like "⌫" or "清除"

### 16. Combo Milestone Animations Fire on Every Increment
```javascript
if ([5, 10, 15, 20].includes(battleState.combo)) {
  display.classList.add('pulse');
  // ...
  screenShake();
}
```
Screen shake fires even when going from 4→5, causing frequent jarring motion.
**Fix**: Consider reducing shake frequency or making it subtler for lower milestones.

### 17. Battle Stats Bar Format Confusion
Line 2925: `🪙 <span id="goldCount">0</span> 分數: <span id="waveScore">0</span>`
- "分數" means "score" in Chinese, but gold and waveScore are different things
- Gold is currency, waveScore is... unclear
**Fix**: Clarify labels or remove waveScore from display.

### 18. Hero Selector Buttons Stay Active After Selection
When hero selected via `selectBattleHero()`, the button gets `.active` class, but after battle starts, selection remains even though battle hero is locked in.
**Fix**: Add visual distinction between "selected" and "currently in battle".

### 19. Accessibility: Missing ARIA for NumPad
Number pad buttons are `<button type="button">` but lack aria-labels for screen readers. Tap targets might also be small for mobile (padding: 18px 0).
