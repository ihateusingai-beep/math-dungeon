# Multiplication Town — Development Roadmap: SP → MP Transition

**Project:** MathBattle ARPG (single-player focused → multiplayer-ready)  
**Horizon:** 6-12 months | **Team:** 1 developer | **MP Types:** Co-op 2-8 players + local hotseat

---

## Phase 0: Architecture Foundation (Month 1-2)

### Goal: Build the game with multiplayer-aware architecture from the start

### 0.1 Code Modularity — Extract Game Logic from UI

**Why:** Single-file HTML (8266 lines) mixes logic, state, and rendering. MP requires sharing state across clients.

**Actions:**
- [ ] Identify core domain objects: `Hero`, `Monster`, `BattleState`, `Question`, `Skill`
- [ ] Create a `GameEngine` class/module that manages pure game logic (no DOM dependencies)
- [ ] GameEngine receives commands (e.g., `answer(question, input)`) and returns state deltas
- [ ] Rendering layer reads from GameEngine state, never mutates it directly
- [ ] Example:
  ```javascript
  class GameEngine {
    state: BattleState  // source of truth
    answer(q, input) → { correct, damage, rewards }
    useSkill(skillKey) → { effect, cooldown }
    // All state mutations go through methods
  }
  ```

**Milestone:** Core game logic is unit-testable without DOM.

### 0.2 State Serialization Layer

**Why:** MP requires sending state over network (co-op) or sharing memory (hotseat).

**Actions:**
- [ ] Add `serialize()` / `deserialize()` to all game entities
- [ ] State should be plain JSON-compatible objects (no DOM references, no closures)
- [ ] Define schema version for future migrations (e.g., `stateVersion: 1`)
- [ ] Example battle state shape:
  ```javascript
  {
    version: 1,
    phase: 'battle', // 'menu' | 'battle' | 'victory' | 'defeat'
    hero: { key, hp, maxHp, mana, level, exp },
    monsters: [{ id, type, hp, maxHp, intent }],
    combo: 0,
    gold: 0,
    shield: 0,
    skillCooldowns: {},
    // For MP: add playerId, sessionId, turnOrder
  }
  ```

**Milestone:** Any state snapshot can be saved/loaded as JSON.

### 0.3 Turn-Based Sync Model

**Why:** This is a turn-based RPG. Turns are natural sync points.

**Actions:**
- [ ] Define `TurnContext`: whose turn, available actions, time limit
- [ ] Actions are discrete: `answer_question`, `use_skill`, `defend`, `item`
- [ ] Each action produces an `ActionResult` with the delta to apply
- [ ] For SP: apply immediately. For MP: broadcast to all clients.

**Milestone:** Turn flow is abstracted so SP and MP share the same code path.

## Phase 1: Core SP Systems with MP Hooks (Month 2-4)

### Goal: Ship playable SP game while laying groundwork for MP

### 1.1 Question & Answer System

**Actions:**
- [ ] `QuestionGenerator` class — generates multiplication questions by difficulty
- [ ] Question types: standard (a × b = ?), missing (a × ? = c), word problems
- [ ] `AdaptiveState` tracks per-player weakness patterns
- [ ] Question objects are serializable: `{ type, operands, answer, hint }`

**MP Hook:** Each player gets own QuestionGenerator with same seed = same question (for competition mode).

### 1.2 Battle System — Phase-Based

**Actions:**
- [ ] Define battle phases: `player_turn` → `answer_phase` → `monster_turn` → `resolution`
- [ ] Each phase has entry/exit hooks — these are where MP sync points occur
- [ ] `player_turn` executes answer → damage calculation → combo update → rewards
- [ ] `monster_turn` executes AI intent (attack/skill/defend) → apply to hero
- [ ] Resolution checks win/lose conditions

**MP Hook:** `onPhaseEnter('player_turn', playerId)` broadcasts "player X's turn" to all clients.

### 1.3 Skill & Combat System

**Actions:**
- [ ] `Skill` objects: `{ key, name, manaCost, cooldown, damage, effect }`
- [ ] Skill effects: damage, heal, shield, buff, debuff
- [ ] `useSkill(player, skillKey)` → validates cooldowns/mana → applies effect
- [ ] Skill unlock tied to hero level (already in code, keep it)

**MP Hook:** Skills broadcast to all clients so they see the same animations.

### 1.4 Hero Progression System

**Actions:**
- [ ] `Hero` class: level, exp, hp, skills, equipment slots
- [ ] `LevelUpTable` — exp thresholds for levels 1-99
- [ ] Progression is per-save-file in SP; per-player in MP

**MP Hook:** Each player has own Hero instance. Party size = 1 (SP) or 2-4 (MP).

### 1.5 Save/Load System

**Actions:**
- [ ] `SaveManager` handles serialization to localStorage
- [ ] Save file format: `{ version, timestamp, heroState, progressState, settings }`
- [ ] Auto-save after each battle turn
- [ ] Export/import save as JSON string (for future cloud save)

**Milestone:** Player progress persists correctly across sessions.

## Phase 2: Local Multiplayer — Hotseat (Month 4-6)

### Goal: Enable 2+ players on same device taking turns

### 2.1 Player Session Manager

**Actions:**
- [ ] `LocalSession` manages multiple players on same device
- [ ] `PlayerProfile`: name, heroKey, hp, gold, exp — all per-player
- [ ] Session can be 1-4 players (expandable to 8)
- [ ] Turn order determined at session start (or random)

**Why hotseat first:** No network infrastructure needed. Tests the shared-state architecture.

### 2.2 Shared Screen Rendering

**Actions:**
- [ ] Render layer receives `SessionState` with current player context
- [ ] UI highlights "Active Player" with distinct color/border
- [ ] Swap player on turn end — same screen, same device
- [ ] "Pass & Play" flow: confirm end of turn → next player's intro screen → begin

**Implementation:**
```javascript
class LocalSessionManager {
  players: PlayerProfile[]
  currentPlayerIndex: number
  advanceTurn() { currentPlayerIndex = (currentPlayerIndex + 1) % players.length }
}
```

### 2.3 Turn Confirmation & Anti-Tamper

**Actions:**
- [ ] End of turn requires explicit button press (prevent accidental turn skip)
- [ ] Answer input only accepted from active player
- [ ] Hero HP/mana is per-player; active player's state drives the HUD

**Milestone:** Two players can complete a full battle on the same device.

## Phase 3: Networked Co-op (Month 6-9)

### Goal: Enable 2-8 players over LAN or internet

### 3.1 Multiplayer Architecture Options

**Option A: Authoritative Server (Recommended for 1 dev)**
- Simple Node.js server holds canonical game state
- Clients send actions, server validates and broadcasts deltas
- Room-based: each battle is a "room" with 2-8 players
- No peer-to-peer = no NAT traversal problems

**Option B: Peer-to-Peer (more complex)**
- WebRTC data channels
- Requires STUN/TURN for NAT traversal
- Higher latency variance

**Recommendation:** Option A. Use WebSocket server (e.g., `ws` npm package or hosted service like PlayFab/Liveblocks).

### 3.2 Network Protocol Design

**Actions:**
- [ ] Define message types: `JOIN_ROOM`, `PLAYER_ACTION`, `STATE_UPDATE`, `TURN_END`, `CHAT`
- [ ] Client sends action only (not state). Server applies and broadcasts.
- [ ] Client interpolation: render immediately for responsiveness, reconcile with server state
- [ ] Latency compensation: show "pending" state, confirm on server ack

**Message format:**
```javascript
{ type: 'PLAYER_ACTION', payload: { playerId, action: 'answer', data: { questionId, input } } }
```

### 3.3 Room/Lobby System

**Actions:**
- [ ] `LobbyService`: create room, join room, leave room
- [ ] Room has: host (can kick), players list, maxPlayers, gameMode
- [ ] Game modes:
  - **Co-op Battle:** All players answer questions together, share HP shield, defeat monsters as team
  - **Versus Race:** First to answer correctly deals damage (competitive)
  - **Relay:** Player A answers, passes to Player B, etc.
- [ ] In-Progress join: late joiners spectate or take over next turn

### 3.4 Co-op Battle Flow

**Why this matters:** Multiplication questions need sync — same question for all players or different questions with shared difficulty check.

**Actions:**
- [ ] **Same Question Mode:** Server generates one question, all players answer. First correct deals damage. All share HP pool.
- [ ] **Relay Mode:** Each player gets own question. Speed determines who deals damage.
- [ ] **Shared HP Pool:** Team has combined HP bar. If any player answers wrong, team takes damage.
- [ ] **Turn Order:** Either simultaneous (all answer at once) or sequential (turn-based).

### 3.5 State Synchronization

**Actions:**
- [ ] Server is authoritative; client renders current known state
- [ ] Client sends actions; server validates → updates state → broadcasts `StateDelta`
- [ ] Client applies delta to local state and re-renders
- [ ] For desync detection: server sends state hash every few seconds; client compares

**Rollback:** If client state diverges, request full state sync from server.

### 3.6 Scaling Considerations

- **Player cap per room:** 8 (fits your requirement)
- **If 8 players, question generation:** stagger difficulty so faster players get harder questions
- **Monster HP scales:** `baseHP * (1 + 0.2 * (playerCount - 1))`
- **Gold/rewards:** shared or per-player configurable per room settings

## Key Architectural Decisions Summary

| Decision | Rationale |
|----------|-----------|
| **GameEngine class isolated from DOM** | Enables testing, and MP can share same engine |
| **State serialized as plain JSON** | Network transfer + future cloud saves |
| **Turn-based as sync point** | Natural boundary for MP state sync |
| **Authoritative server for co-op** | 1 dev can manage; avoids P2P NAT complexity |
| **Room-based (not persistent world)** | Simpler; each battle is self-contained session |
| **Hotseat before networked** | Validates architecture with no infrastructure cost |

## Risk Mitigation

| Risk | Mitigation |
|------|------------|
| Architecture too complex for 1 dev | Hotseat first validates shared-state model with zero networking |
| Network synchronization bugs | Use turn-based (not real-time) — simpler to debug |
| Performance at 8 players | Profile early; render optimization is independent of MP |
| Breaking existing SP saves | Version schema; migration function on load |
